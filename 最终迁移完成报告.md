# 最终迁移完成报告

## 概述

本次迁移的目标是把项目从“Rust 外壳 + 仓库内置 Lua backend 资产”重构为“纯 Rust 工程 + 通过 `get_lua_state` 访问 Cheat Engine 原生能力”的实现形态。

截至当前版本，迁移已经完成，仓库不再依赖独立的 Lua 文件资产来承载正式运行时能力。

## 迁移目标达成情况

### 已达成

- 删除仓库内正式运行所需的 Lua 资产文件
- 删除历史 `lua_backend.rs` / `lua_client.rs` 桥接层
- 保留并强化 `lua_host.rs`，通过 `get_lua_state` 访问 CE 原生能力
- 核心工具链迁移到 CE-native runtime bridge
- 保持 MCP 对外工具面基本稳定
- 增加关闭插件时的 CE 运行时状态清理
- 增加可配置的调试文件日志能力

### 明确未采用的方案

- 未退回 WinAPI 业务后端
- 未把核心能力改写为独立于 CE 的外部实现
- 未保留双运行时形态（Rust + 仓库 Lua 文件）

## 关键架构变化

### 旧架构

```text
Rust tools
  -> lua_client.rs
    -> lua_backend.rs
      -> ce_plugin/src/lua/*.lua
        -> Cheat Engine Lua globals / CE objects
```

### 新架构

```text
Rust tools
  -> lua_host.rs
    -> get_lua_state
      -> 内联 Lua 片段 / Lua C API
        -> Cheat Engine Lua globals / CE objects
```

## 运行时清理能力

为避免插件重载或关闭后残留 CE 全局状态，新增了统一清理入口：

- 内部方法：`__ce_mcp_cleanup_runtime_state`
- Rust 侧入口：`crate::tools::cleanup_ce_runtime_state()`
- 调用时机：`shutdown_runtime()`

当前会清理：

- `_G.__ce_mcp_breakpoints`
- `_G.__ce_mcp_breakpoint_hits`
- `_G.__ce_mcp_hw_bp_slots`
- `_G.__ce_mcp_active_watches`
- `_G.__ce_mcp_scan_foundlist`
- `_G.__ce_mcp_scan_memscan`

并尝试：

- 删除硬件断点
- 停止 DBVM watch
- 销毁 scan foundlist / memscan

## 调试日志能力

配置新增：

```json
{
  "runtime": {
    "debug_enabled": true
  }
}
```

### 行为说明

- `debug_enabled = true` 时：
  - 保留控制台日志
  - 在插件 DLL 同级目录落地 `ce_plugin.debug.log`
- 控制台与落地日志统一使用简体中文输出
- 日志内容用于排查 MCP 请求、CE runtime bridge、脚本调用和运行时状态问题
